# 排班系統

這個專案是排班管理的網頁版程式，主要功能包含：

- 班表編輯
- 班別、假別、加班設定
- 人員、單位設定
- 休假與加班申請
- 例休檢查
- 匯入 / 匯出

目前前端來源在 `src/renderer/`，GitHub Pages 發佈內容在 `docs/`。

## 啟動與發佈

常用指令：

```bash
npm run web
npm run web:check
npm run web:publish
```

說明：

- `npm run web`：啟動本機網頁版
- `npm run web:check`：檢查公開版 Supabase 設定
- `npm run web:publish`：把 `src/renderer/` 產生到 `docs/`，供 GitHub Pages 使用

只要有改到網頁畫面、互動、樣式或前端資料流程，就要重新執行 `npm run web:publish`，否則 GitHub Pages 不會更新。

## 目前資料結構重點

- `src/renderer/`
  - 前端畫面、互動、班表邏輯
- `docs/`
  - GitHub Pages 靜態輸出
- `supabase/`
  - SQL schema、migration、Edge Functions
- `scripts/`
  - 發佈與檢查腳本

## 自動排班相關狀態

目前還沒有正式的「自動排班演算法」，但已先把之後需要的基礎設定補好，方便下一步接自動排班。

### 已完成的基礎設定

1. 月週設定

- 浮動功能視窗中的 `每週起算設定` 已改成 `月週設定`
- 目前可設定：
  - `每週起算日`
  - `每月起算日`
- 班表週期格線與例休檢查會依每週起算日連動
- 每月起算日會作為之後計算「月休天數」與自動排班月區間的基準

2. 班別需求人數

- 在修改班別頁新增 `需求人數`
- 用來表示每個單位在某個班別下需要幾位人員
- 這是之後自動排班補人時的基礎資料

3. 人員排班單位

- 修改人員頁中的 `所屬單位` 概念已延伸為 `排班單位`
- 可勾選多個可支援單位
- 可拖曳排序
- 第一個單位視為主要單位，也就是原本的所屬單位
- 後面的單位代表可支援單位，排序越前面優先度越高

4. 月休天數

- 修改人員頁新增 `月休天數`
- 定義為該月內 `例假 + 休息日` 的目標天數
- 月的切分方式以 `月週設定` 的每月起算日為準

5. 單位設定雙檢視

- 單位設定頁目前可切換：
  - `單位檢視`
  - `人員檢視`
- 單位檢視會看每個單位的：
  - 所屬人員
  - 可排人員
- 人員檢視會看每位人員的：
  - 排班單位排序結果

### 自動排班預定規則

目前已整理出的方向如下：

- 一人一天最多一班
- 優先排本單位，不足時才依排班單位順序支援外單位
- 主管先手動指定的班與假，視為鎖定不動
- 已指定的休假日數會納入月休天數計算
- 目標是固定月休天數，不是至少幾天
- 若排不滿，先保留空班，不強行破壞既有指定
- 假日安排希望讓每人休六日的機會盡量平均
- 同一天同單位若多個班缺人，先依班別順序補

## 例休檢查

目前例休檢查重點：

- 每 7 天至少 1 天例假
- 每 7 天至少 1 天休息日
- 連續出勤不得超過 6 天

補充：

- `連續出勤 > 6 天` 是用任意滑動日數檢查，不是只看單週
- 檢查會包含上個月銜接資料
- 目前系統以已標記的 `例假 0036 / 休息日 0047` 為準
- 空白未排班不自動視為例假或休息日

## 資料庫 / Supabase

自動排班基礎設定目前已對應到資料欄位，重點包含：

- `profiles.schedule_department_ids`
- `profiles.monthly_rest_days`
- `member_departments.sort_order`
- `shift_types.required_staff_count`
- `schedule_months.month_start_day`

如果新環境要補齊這些欄位，請確認有執行：

- `supabase/015_auto_schedule_settings.sql`

如果有動到會員 / 人員同步相關欄位，也要同步確認：

- `supabase/functions/member-auth-admin/index.ts`
