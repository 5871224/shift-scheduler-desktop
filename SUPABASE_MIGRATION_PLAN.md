# 排班工具改網頁版 + Supabase 規劃

## 目前共識

- 目標改為 `網頁版 + Supabase`
- 排班資料存在資料庫
- 匯出 Excel 為當下產生後直接下載，不長期存雲端
- 不上傳大附件
- 第一階段先不使用 Supabase Storage

## 目前本機版資料現況

- 目前 SQLite 只有一張 `app_state` 表
- 主要資料是整包 JSON 存在 `payload`
- 目前不是正規化成多張資料表

## 建議遷移方向

第一階段先走最小可行版本：

1. 先做網頁版 + Supabase Database
2. 先把目前整包排班 JSON 存到 Supabase
3. 先不要急著拆很多表
4. 等流程穩定後，再逐步正規化

原因：

- 改動最小
- 上線最快
- 目前需求還在持續調整
- 先求可用，再求漂亮的資料表結構

## 角色與權限規則

### 員工

- 可以看所有人的班表
- 可以用手機上下班打卡
- 可以申請請假
- 可以申請加班

### 主管

- 包含所有員工權限
- 可以管理多個單位
- 可以做單位設定
- 可以做人員設定
- 可以做班別設定
- 可以調整班表
- 可以審核員工請假
- 可以審核員工加班
- 已審核的請假和加班仍可修改

## 業務規則

- 一個主管可以管理多個單位
- 員工會跨單位支援
- 員工可以看到所有人的班表
- 打卡有地點限制
- 打卡時間要對應班表，用來判斷是否遲到早退

## 建議的第一版資料設計

第一版先建一張主表即可，例如：

`schedule_documents`

欄位概念：

- `id`
- `name`
- `payload`
- `updated_at`
- `user_id`

說明：

- `payload` 先存目前完整排班 JSON
- `user_id` 用來對應登入者
- 後續若要多人共用、版本控管、歷史資料，再擴充欄位

## 為什麼先不要急著拆很多表

如果一開始就拆成：

- 單位表
- 人員表
- 班別表
- 假別表
- 加班表
- 班表表
- 請假申請表
- 加班申請表
- 打卡表

雖然結構會比較完整，但現在需求還在調整，開發成本會大很多。

## 登入方式結論

不建議自己做帳密表。

建議：

- 使用 Supabase Auth 做登入
- 我們自己的資料表只管角色、單位、排班資料

原因：

- 比較省工
- 比較安全
- 後續權限控制比較簡單

## 手機打卡建議

第一版先做：

1. 手機開網頁
2. 允許定位
3. 抓目前 GPS
4. 比對是否在允許打卡地點半徑內
5. 在範圍內才允許打卡
6. 再對照班表判斷正常、遲到、早退

### 打卡要記錄的資料

- 打卡時間
- 員工
- 打卡類型（上班 / 下班）
- GPS 緯度經度
- 對應地點
- 與地點距離
- 判斷結果（正常 / 遲到 / 早退 / 異常）

### 打卡地點限制注意事項

- 網頁版可以限制地點，但不是百分之百無法繞過
- 需要 HTTPS 才能使用瀏覽器定位
- 員工必須允許手機定位權限
- 定位會有誤差，半徑不能設太死

### 建議第一版打卡半徑

- 可先從 `100 公尺` 開始

## 建議的實作順序

1. 建立 Supabase 專案
2. 建登入
3. 建人員資料與角色
4. 建單位 / 班別 / 假別 / 加班別設定
5. 建班表
6. 建請假申請與審核
7. 建加班申請與審核
8. 建打卡功能

## 已確認的補充規則

- 每個班別只會對應一個適用單位
- 因此班別設定可直接把適用單位放在班別表，不需要另外拆一張班別對單位對照表
